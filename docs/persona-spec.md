# Persona Interface Specification

This document defines what a persona needs to provide to be compatible with openCouncil.

## Minimum Viable Persona (Inline)

An inline persona is a 2-5 sentence string that gives the AI enough to simulate a distinct perspective. It must contain:

1. **Worldview** — one sentence on what this persona optimizes for
2. **Mental models** — 1-3 named thinking frameworks
3. **Signature question** — what this persona always asks

### Example

```
You optimize for shipping speed and reducing complexity.
Mental models: YAGNI, time-to-value, opportunity cost.
You always ask: "What's the simplest version we can ship this week?"
```

This is enough for Mode A (Quick Discussion). For deeper fidelity, use external persona files.

---

## Full Persona File (External)

A full persona file is a markdown document (typically a SKILL.md) that the AI reads before speaking as that character. openCouncil is compatible with any persona file that contains the following three sections, in any format:

### 1. Identity

Who this persona is. Their name, background, and what they're known for.

```markdown
## Identity

Shigeru Miyamoto. Nintendo's lead game designer for 40+ years.
Creator of Mario, Zelda, Donkey Kong. Known for polishing gameplay
feel until non-gamers can pick up and play.
```

### 2. Mental Models

How this persona thinks. Named frameworks, principles, or heuristics they apply to problems. This is the most important section — it's what makes the persona's opinions *distinct* from generic AI output.

```markdown
## Mental Models

1. **Wife Test** — If my wife can't understand it without instructions, it's too complicated.
2. **Progressive Complexity** — The first 5 minutes teach through play, not text.
3. **Upending the Tea Table** — When something isn't working, throw it all out and start over.
4. **Garden Design** — Guide the player's attention like paths in a garden.
```

Aim for 3-6 mental models. Fewer feels thin. More than 8 dilutes focus.

### 3. Expression DNA

How this persona speaks. Their tone, verbal patterns, and the kinds of phrases they use.

```markdown
## Expression DNA

- Speaks simply and concretely. Uses physical-world metaphors.
- Never says "optimize" or "leverage." Says "feel" and "surprise."
- Asks questions more than making statements.
- Frequently references what he observed real people doing, not what theory predicts.
- Signature move: demonstrates a point with a tiny physical interaction example.
```

## Optional Sections

These sections aren't required but improve council performance:

### Domain Boundaries

What the persona knows well and where they'll defer.

```markdown
## Domain Boundaries

**Strong:** game feel, UX, controller design, level design, family entertainment
**Weak:** server architecture, monetization math, ad tech, competitive esports meta
```

This helps openCouncil's "honest boundaries" rule — personas with explicit boundaries will correctly say "this isn't my area" instead of fabricating opinions.

### Decision Heuristics

Concrete rules the persona uses when making choices.

```markdown
## Decision Heuristics

1. If the player needs a tutorial popup, the design has failed.
2. If a feature is fun alone but breaks the flow, cut it.
3. When in doubt, prototype with paper before touching code.
```

### Weight Topics

Keywords indicating which discussion topics this persona should lead on.

```markdown
## Weight Topics

ux, game-feel, onboarding, accessibility, controls, level-design
```

---

## Compatibility Notes

- openCouncil reads persona files as plain text. No special YAML frontmatter is required (but it's tolerated).
- The file can contain additional sections beyond the three required ones. openCouncil will use whatever context is available.
- Persona files generated by [Nuwa Skill](https://github.com/alchaincyf/nuwa-skill) are fully compatible — they already contain Identity, Mental Models, and Expression DNA.
- If a persona file is very long (>2000 lines), the AI may not internalize all of it. Consider creating a council-optimized summary version.

---

## Persona Quality Checklist

Before adding a persona to your council, verify:

- [ ] Can you predict what this persona would say about a random topic? (If not, the definition is too vague)
- [ ] Are the mental models distinct from the other council members? (If two personas apply the same frameworks, merge them)
- [ ] Does the expression DNA produce recognizably different speech? (If two personas sound the same, sharpen their voices)
- [ ] Are domain boundaries defined? (Prevents persona from opining on everything)
- [ ] Would this persona disagree with at least one other council member on a typical question? (If everyone always agrees, the council is pointless)
